## Open Questions

These are the live technical unknowns the product must resolve. Several now have concrete, evidence-backed answers; the rest remain genuinely open.

**1. Recovering PDF heading depth (bookmarks + alphanumeric numbering, not font size) — LARGELY ANSWERED.**
- **`docling-hierarchical-pdf` (krrome) is the direct answer for the Docling route** and is **production-installable today** (`pip install docling-hierarchical-pdf`, current PyPI v0.1.8, released 2026-04-24). It is a post-processing add-on: run Docling's `converter.convert(source)`, then `ResultPostprocessor(result).process()`, which corrects `result.document` hierarchy in place. It recovers depth via **three layered strategies in priority order**: (1) PyMuPDF-extracted PDF bookmark/outline TOC, (2) heading-numbering inference (Arabic, Roman, and letter/outline numbering — directly handles "A.1.1", "B4-3.2"-style prefixes), (3) font-size/style (bold/italic) only as a fallback. **[high]** — github.com/krrome/docling-hierarchical-pdf; pypi.org/project/docling-hierarchical-pdf/
- **For the PyMuPDF route, `pymupdf4llm.TocHeaders` seeds heading levels from `Document.get_toc()`** (the embedded bookmark outline) instead of font size — this is the documented "outline seeding" technique. **[high]** — pymupdf.readthedocs.io/en/latest/pymupdf4llm/api.html
- **For numbering-prefix → depth heuristics under PyMuPDF**, the documented extension point is `to_markdown(..., hdr_info=callable)`: a user callable `(span, page=None)` returning `""` or up to six `#` + space. You can implement alphanumeric-numbering → depth logic here. **[high]** — same API page.
- **Caveats:** `TocHeaders` only works when a real outline TOC exists, assumes bookmark text appears verbatim as a heading line on the page ("which certainly need not be the case"), and is unavailable in layout mode. The default `IdentifyHeaders` is purely font-size driven and is what produces the flattening this product is fighting. **DECISION:** prefer outline-TOC seeding first, numbering-prefix heuristic second, font-size last — mirror the `docling-hierarchical-pdf` priority order regardless of which extractor you adopt.

**2. pymupdf4llm over-segmentation (bold inline runs → spurious headings) — PARTIALLY ANSWERED.** The root cause is confirmed: the default `IdentifyHeaders` maps any font size larger than the most-popular body size to `h1–h6`, so bold/large inline runs get promoted. The documented mitigations are (a) switch to `TocHeaders` (outline-driven, ignores inline font size entirely) or (b) supply a custom `hdr_info` callable that gates on numbering prefixes / line position, not just size. **[high]** — pymupdf.readthedocs.io/en/latest/pymupdf4llm/api.html. **STILL OPEN:** no verified config flag that simply suppresses single-span bold-run promotion; a post-filter (drop "headings" whose text length / line geometry looks like inline emphasis) is likely required and is unverified.

**3. Unicode/ligature artifacts (ﬂ/ﬁ, intra-word spaces) — MECHANISM CONFIRMED, NORMALIZATION POLICY OPEN.** PyMuPDF **preserves ligatures by default** (`TEXT_PRESERVE_LIGATURES` is "on"), passing combined glyphs ('ffi', 'fi', 'fl') through unchanged. To split them you must explicitly clear that bit in the `flags` bitmask passed to `get_text` (note: passing `TEXT_PRESERVE_LIGATURES=0` as a standalone kwarg does NOT work — GitHub issue #375). MuPDF handles 7 ligatures (ff, fi, fl, ffi, ffl, ft, st). **[high]** — pymupdf.readthedocs.io/en/latest/app1.html, vars.html. **DECISION DRAFT:** either clear the ligature flag at extraction OR apply NFKC normalization downstream (NFKC decomposes these ligatures to ASCII). **STILL OPEN:** intra-word spurious spaces are a separate defect not addressed by ligature handling or NFKC and need a dedicated heuristic/dictionary pass — unverified.

**4. AGPL-3.0 exposure of PyMuPDF / pymupdf4llm for a network-served product — TRIGGER CONDITION CONFIRMED.** pymupdf4llm is **AGPL-3.0** (LICENSE = "GNU AFFERO GENERAL PUBLIC LICENSE, Version 3, 19 November 2007"), dual-licensed AGPL-3.0 **OR** Artifex commercial. **[high]** — github.com/pymupdf/pymupdf4llm/blob/main/LICENSE; PyPI classifier. AGPLv3 §13's network-source-offer duty is triggered **only when you run a *modified* version** reachable by remote users: "your modified version must prominently offer all users interacting with it remotely through a computer network … an opportunity to receive the Corresponding Source." **[high]** — same LICENSE; SPDX AGPL-3.0; gnu.org/licenses/agpl-3.0; opensource.com legal analysis. **DECISION:** (a) if you ship pymupdf4llm *unmodified* behind the MCP/FastAPI service, §13's modification trigger is arguably not hit — but "modification" is the one genuinely disputed term in the literature, and the parsing library's network exposure is supplied by your FastAPI/MCP wrapper, so treat this as legal risk to clear, not a settled safe-harbor; (b) the clean options are an Artifex commercial license **or** migrating to MIT-licensed Docling. **STILL OPEN / NOT FOUND:** a verified Artifex commercial-license price ballpark — do not state a number without a quote.

**5. MIT alternative — Docling (IBM).** Docling + `docling-hierarchical-pdf` is the permissively-licensed escape from AGPL and is the heading-depth recovery answer in #1. **[high]** for the add-on's existence/behavior. **STILL OPEN (thin evidence):** verified runtime-footprint numbers (PyTorch + DocLayNet/TableFormer model download sizes, Docker image size, CPU inference latency) and model-cache-at-build-time guidance were not independently verified in this pass — flag as a benchmarking task before committing to Docling in the hot path.

**6. Tables inside a heading-tree RAG (pipe-table-as-node-text vs structured leaf nodes) — OPEN.** No verified primary evidence on which representation yields better tree-search retrieval quality. Decision deferred; prototype both against the validation corpus.

**7. Cross-document graph layer (lightweight networkx edges vs Microsoft GraphRAG / LazyGraphRAG) — PARTIALLY GROUNDED.** General principle from the tree-RAG literature: short-chunk retrieval limits holistic, cross-document understanding, motivating structure that spans documents. **[high]** — arxiv 2401.18059 (RAPTOR). But **the specific threshold** (when a `<10`-doc networkx 'contains'/'variant-of'/'amends' graph suffices vs when full GraphRAG is justified) was **not** established by a verified source. Heuristic to validate, not cite: single-hop/within-family lookups → networkx is adequate; global multi-hop synthesis across a large corpus → consider GraphRAG. STILL OPEN.

**8. Document versioning / supersedes semantics — PARTIALLY GROUNDED via erasure cascade.** The strongest verified adjacent evidence is that derived artifacts form a multi-store cascade (see Compliance / claim on AWS Bedrock RTBF). That argues for **content-hash dedup as the identity key** (so reissued documents are detected) plus explicit `effective_date` / `doc_family` / "as-of-date" retrieval metadata. STILL OPEN: no verified canonical pattern for supersedes-chains specific to tree-RAG; design from first principles.

**Additional open questions that emerged:**
- Is shipping pymupdf4llm *unmodified* genuinely outside AGPL §13, or does the FastAPI/MCP wrapper constitute a combined/modified work? (legal interpretation, unresolved)
- What is the real-world reliability rate of embedded bookmark outlines across arbitrary corpora (insurance, legal, scientific) — i.e., how often does the TOC-first strategy actually fire? (no corpus-level measurement found)
- Verified Docling CPU-inference latency and image-size budget for the ingest worker. (benchmarking gap)
- An automated, validated quality gate threshold (node count / depth / garbling) for "good tree" — proposed below but not empirically calibrated.

## Domain

**Vectorless / tree-reasoning RAG vs embedding RAG — evidence is THINNER than vendor marketing implies; be skeptical.**
- The *motivation* for hierarchical/tree retrieval is solid and primary-sourced: standard RAG "retrieve[s] only short contiguous chunks … limiting holistic understanding of the overall document context," which is why tree methods that "integrate information across … different levels of abstraction" exist. **[high]** — arxiv 2401.18059 (RAPTOR, ICLR 2024).
- **Crucial distinction for positioning:** RAPTOR is *tree-organized* but **embedding-dependent** — it recursively *embeds, clusters (GMM+UMAP), and summarizes* chunks, and traverses via cosine similarity (SBERT `multi-qa-mpnet-base-cos-v1`). It is **not** the same as PageIndex's purely-vectorless, reasoning-based tree search. **[high]** — arxiv 2401.18059; Stanford CS224N report; Superlinked VectorHub. Use RAPTOR as the "tree-but-still-vector" contrast point, not as evidence for vectorless retrieval.
- **CRITICAL CAVEAT — the headline "vectorless beats vector RAG" benchmark claims did NOT survive verification.** Three separate quantitative claims sourced to arxiv 2604.14222 were **REFUTED** by adversarial review (votes 0-3, 1-2, 0-3): that tree/hybrid hit 100% cross-reference recall vs 91.7% vector; that tree reasoning dominates legal (0.883)/medical (0.933) while vector wins finance; and that tree RAG (0.938) beats hybrid/vector on FinanceBench. **Do NOT cite accuracy superiority numbers for vectorless RAG in the PRD** — the supporting source did not hold up. Position the vectorless approach on its *architectural* merits (no vector DB to operate, transparent/inspectable tree, structural-query alignment, lower infra surface), not on unverified accuracy wins. **[high]** that these specific numbers are unreliable.

**Born-digital vs scanned, tagged PDF, and the bookmark-outline oracle.** The product's hierarchy-recovery strategy leans on embedded bookmark/outline TOCs (claims in Open Questions #1). The verified tooling treats the outline as the *first-choice* oracle but every primary source flags it as conditional — present only sometimes, and bookmark text need not match on-page heading text. **[high]** (docling-hierarchical-pdf; pymupdf4llm TocHeaders docs). **Evidence gap:** no verified cross-corpus measurement of outline reliability, and multi-column reading-order reconstruction / DocLayNet layout-model state-of-the-art was **not** independently verified in this pass — treat both as design assumptions to validate, not established facts.

**Actionable quality-gate thresholds for a "good" extracted tree (PROPOSED, not empirically calibrated — flag in DESIGN).** Derived from failure modes the verified sources expose, not from a benchmark:
- **Depth > 1 required** when an outline TOC or numbering prefixes exist (a tree that is all-`h2` indicates the font-size flattener fired and the outline/numbering path was missed). Gate on: `max_depth >= 2` for any doc with a non-empty `get_toc()`.
- **Garbling heuristics:** flag nodes whose text contains residual ligature glyphs (ﬁ/ﬂ/ﬃ) post-extraction, or an abnormal intra-word-space ratio — both are confirmed PyMuPDF artifacts (Open Questions #3).
- **Over-segmentation heuristic:** flag when heading-node count is implausibly high relative to page count (symptom of bold-run promotion, Open Questions #2).
- These are starting thresholds for an automated validation gate; **calibrate against the German-insurance corpus before enforcing in CI.**

## Compliance

This is a document-*processing* service: arbitrary user uploads may contain PII; documents are stored in MinIO and sent to third-party LLM APIs. The verified findings give clear, citable controls.

**Right-to-erasure must cascade across EVERY derived store — this is the single most load-bearing compliance finding.** In a RAG knowledge base, erasure must purge, separately: the **source object(s)** (MinIO/S3 — here `uploads/`, `preloaded/`), the **derived index** (here the `processed/*.json` trees + `*.meta.json` sidecars), **session/derived history** (the Redis cache; any job records), and **backups**. **Backup purging is explicitly the customer/operator's responsibility and is NOT automatic.** **[high]** — aws.amazon.com/blogs/machine-learning/…right-to-be-forgotten… **DESIGN IMPLICATION:** a DSR/erasure operation in this product must fan out to MinIO (`uploads/`, `preloaded/`, `processed/<id>.json`, `processed/<id>.meta.json`), invalidate the Redis cache entry, AND have a documented manual backup-purge step. **CAVEAT:** the related claim that "source-file deletion auto-cascades to derivatives" was **REFUTED** (vote 1-2) — do NOT assume deleting the raw blob removes the derived tree/cache; purge each store explicitly.

**Third-party LLM API data-handling — all three major providers offer no-training-by-default + a zero-retention path + EU residency / DPA. These are the residency escape hatches.**

- **OpenAI:** By default does **not** train on API / business data (API Platform after 2023-03-01, ChatGPT Business/Enterprise/Healthcare/Edu/Teachers) unless explicitly opted in. **[high]** — openai.com/enterprise-privacy. Offers **Zero Data Retention (ZDR)** for eligible endpoints to qualifying customers (prompts/outputs excluded from abuse-monitoring logs / `store=false`); documented exceptions exist (CSAM review, ~1h audio state, background-mode ~10min, prompt-cache tensors). **[high]** — developers.openai.com/api/docs/guides/your-data. Will **execute a GDPR DPA** for Business/Enterprise/API. **[high]** — openai.com/enterprise-privacy + openai.com/policies/data-processing-addendum (executed DPA artifacts exist in the wild). Offers **EU data residency for the API Platform** (announced 2025-02-05) — but only for **new** Projects; existing Projects can't migrate, and residency covers only eligible endpoints. **[high]** — openai.com/index/introducing-data-residency-in-europe.

- **Anthropic (Claude API):** Offers **ZDR** — "Customer data is not stored at rest after the API response is returned, except where needed to comply with law or combat misuse." Applies to **Claude Messages + Token Counting APIs** (and Claude Code with Commercial-org API keys / Claude Enterprise); **NOT** to Console/Workbench, consumer Free/Pro/Max, or third-party integrations. **[high]** — platform.claude.com/docs/en/manage-claude/api-and-data-retention; privacy.claude.com. Retained data is "never used for model training without your express permission"; conversation content "never retained unless explicitly noted." **[high]** — same. (Caveat: explicit retention exceptions exist — Batch 29d, Files until-deleted, code-exec 30d, policy-violation up to 2y. The 2025 consumer opt-out-or-train change is scoped to consumer products, NOT the API.)

- **Azure OpenAI / Foundry Models:** Prompts, completions, embeddings, and training data are **NOT** used to train foundation models without permission, **NOT** available to OpenAI/other providers, **NOT** available to other customers. **[high]** — learn.microsoft.com/en-us/azure/foundry/responsible-ai/openai/data-privacy (current, ms.date 2026-05-18). Default abuse monitoring stores a sampled subset up to 30 days for authorized-Microsoft human review *within the service boundary*. **Managed customers can apply for "modified abuse monitoring"** which turns off that storage + human review (effectively a zero-retention path), verifiable via `ContentLogging=false` in the Azure portal / `az cognitiveservices account show`. **[high]** — same. For **EU data residency**, abuse-monitoring data is stored in the Azure geography where the Foundry resource is deployed, and **for EEA deployments the human reviewers are located in the EEA**. **[high]** — same. (Caveat: these are contractual/policy commitments, not independently audited technical guarantees; protections are conditional on the customer not opting into fine-tuning/stored completions.)

**Synthesis for the product:** route PII-bearing documents through a provider configured for (a) no-training default — true by default on all three — (b) ZDR / modified-abuse-monitoring enabled, and (c) EU residency where the corpus warrants it (insurance/financial/health expectations lean EU-hosted). The `OPENAI_BASE_URL` indirection already in the repo is the lever to point at Azure-EU or a **self-hosted/local model**, which is the ultimate residency escape hatch when no cloud tier is acceptable. **[high]** on each provider control; **[medium]** that local-model self-hosting is the right fallback (architectural inference, not a single cited source). **Sector note:** sending insurance/financial/health documents to cloud APIs raises data-residency expectations — verified provider EU-residency/DPA options above are the compliant mechanisms; specific sector-regulatory minimums were not independently verified and should be confirmed per deployment jurisdiction.

## Market & Design

This is a backend/MCP product — "design" = API/DX/operability, not UI.

**Competitor landscape — positioning, not benchmark superiority.** The document-parsing + RAG-ingestion field includes LlamaParse, Unstructured.io, Docling, Azure AI Document Intelligence, AWS Textract, Reducto, and PageIndex's own cloud OCR. **Verified, defensible differentiation for a self-hosted vectorless-tree approach:** (a) **no vector DB to operate** — lower infra surface than embedding-RAG competitors; (b) **transparent, inspectable hierarchical trees** as the retrieval substrate (vs opaque embedding spaces); (c) **structural-query alignment** for documents whose meaning lives in hierarchy (insurance T&C, legal, contracts). **DO NOT differentiate on accuracy/benchmark wins** — the three quantitative "vectorless beats vector" claims were **REFUTED** in verification; that ground is unsupported. **[high]** that the accuracy-superiority pitch is unsupported; **[medium]** on the architectural differentiation framing (reasoned from verified RAPTOR/tree-RAG motivation + the no-embeddings design, not from a competitor teardown source). Detailed per-competitor pricing/strength/gap tables were **not** independently verified — flag as a market-research gap.

**MCP-server DX/operability conventions (grounded in this product's own surface + verified RAG-ops principles):**
- **Tool naming / structured outputs:** the existing tool set (`recent_documents`, `find_relevant_documents`, `get_document`, `get_document_structure`, `get_page_content`) is already verb-noun, query-shaped, and machine-consumable — keep that convention.
- **Async-job status polling:** the existing `POST /upload/files` → `GET /upload/status/{job_id}` pattern is the right idempotent-ingest + poll convention; pair with content-hash dedup (Open Questions #8) so re-uploads are idempotent.
- **Erasure as a first-class operation:** because erasure must cascade across MinIO + Redis + backups **[high]** (Compliance), expose/operate a DSR purge that fans out to all derived stores — do not rely on raw-file deletion cascading (REFUTED).
- **Observability:** Prometheus metrics + health endpoints are in scope and consistent with standard practice (not separately verified here).

**Accessibility/a11y = machine-consumability of outputs.** For this product, a11y means downstream agents get **clean structured trees with correct depth and faithful tables**. The heading-depth-recovery work (Open Questions #1–#2) and ligature/normalization work (#3) are *directly* a11y work: a flattened, garbled, or over-segmented tree is an inaccessible output for an LLM consumer. **[high]** on the underlying extraction-defect facts; **[medium]** on framing them as the a11y axis.
